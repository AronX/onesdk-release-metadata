# Metadata Visual Admin 交互方案

## 1. 交互目标

这套后台的交互目标不是“让用户编辑 JSON”，而是“让用户维护 OneSDK 与渠道 SDK 的映射关系”。

因此交互设计要解决 4 件事：

1. 先看清当前有哪些 OneSDK 版本
2. 再看清某个版本下有哪些渠道和渠道版本
3. 用表单而不是直接改 JSON 的方式完成新增和编辑
4. 在保存前明确告诉用户改了什么、有没有风险

## 2. 使用场景

主要场景只有 3 个：

### 场景 A：查看当前映射

用户进入后台后，先确认：

- 当前有哪些 OneSDK 版本
- 某个 OneSDK 版本下是否已有某个渠道
- 某个渠道下是否已有某个渠道版本

### 场景 B：新增 OneSDK 版本

用户要新增一个 OneSDK 版本，例如：

- `1.37.0`

此时系统需要：

- 创建新的版本文件
- 更新 `index.json`
- 让用户立刻进入该版本的空白详情页

### 场景 C：新增或修改渠道版本条目

用户在某个 OneSDK 版本下：

- 新增 `bilibili` 或 `lumo`
- 新增某个渠道版本条目
- 修改已有条目的 URL、MD5、发布日期、更新说明

## 3. 页面结构

推荐采用桌面端优先的三段式布局：

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部工具栏                                                  │
├──────────────┬──────────────────────────────┬──────────────┤
│ 左侧版本栏    │ 中间详情区                     │ 右侧编辑区     │
│              │                              │              │
│ OneSDK列表    │ 当前版本 / 渠道 / 条目表格       │ 抽屉或侧栏表单  │
│              │                              │              │
└──────────────┴──────────────────────────────┴──────────────┘
```

### 3.1 顶部工具栏

顶部工具栏固定展示：

- 页面标题：`OneSDK Metadata Admin`
- 当前仓库状态：
  - 已加载
  - 有未保存修改
  - 保存失败
- 按钮：
  - `新增 OneSDK 版本`
  - `刷新`
  - `保存`
  - `查看变更`

### 3.2 左侧版本栏

左侧只承担“切版本”的职责，不承载编辑。

每个版本卡片展示：

- OneSDK 版本号
- OneSDK 发布日期
- 渠道数量
- 是否有未保存修改标记

交互规则：

- 单击切换当前版本
- 当前选中项高亮
- 如果当前版本有未保存修改，切换前弹确认

### 3.3 中间详情区

中间是最核心的信息区，分 3 段：

1. 版本头部信息
2. 渠道分组信息
3. 渠道版本条目表格

#### 版本头部信息

展示：

- OneSDK 版本号
- 版本发布日期
- 版本文件路径
- 按钮：
  - `编辑版本信息`
  - `新增渠道`

#### 渠道分组信息

每个渠道以分组卡片展示：

- 渠道名，例如 `bilibili`
- 条目数
- 按钮：
  - `新增版本条目`

#### 渠道版本条目表格

每个渠道下面用表格展示条目，列建议如下：

- 渠道版本号
- 渠道发布日期
- MD5
- ZIP 地址
- 更新说明摘要
- 操作

操作列按钮：

- `编辑`
- `复制`
- `删除`

其中：

- `复制` 用于快速创建相近版本条目
- `删除` 只删除当前版本文件中的该条记录，不影响其它版本

### 3.4 右侧编辑区

右侧编辑区建议采用抽屉或固定侧栏，不跳页。

编辑对象有 3 种：

1. OneSDK 版本
2. 渠道
3. 渠道版本条目

根据不同对象切换表单内容。

## 4. 关键交互流程

## 4.1 首次进入页面

### 目标

让用户在 3 秒内知道：

- 系统是否加载成功
- 当前有哪些 OneSDK 版本
- 默认打开的是哪个版本

### 流程

1. 页面加载
2. 顶部显示 `正在加载 metadata...`
3. 调用 `GET /api/index`
4. 成功后左侧渲染版本列表
5. 默认打开最新版本
6. 再调用 `GET /api/version/:version`
7. 中间渲染版本详情

### 异常状态

如果 `index.json` 读取失败：

- 页面显示错误态卡片
- 提供 `重试` 按钮
- 不展示空白表格，避免误导用户以为“没有数据”

## 4.2 新增 OneSDK 版本

### 入口

- 顶部工具栏 `新增 OneSDK 版本`

### 交互形式

弹出模态框，而不是右侧抽屉。

原因：

- 这是全局动作
- 不是编辑当前条目，而是创建新的主对象

### 字段

- `OneSDK 版本号`
- `发布日期`

### 校验

- 版本号必须符合 `x.y.z`
- 不能与现有版本重复
- 发布日期不能为空

### 成功结果

保存成功后：

1. 自动创建 `mappings/versions/<version>.json`
2. 自动更新 `index.json`
3. 左侧列表新增该版本
4. 自动切换到新版本详情页
5. 中间区显示空状态，引导用户新增渠道

## 4.3 新增渠道

### 入口

- 当前版本头部 `新增渠道`

### 交互形式

右侧抽屉表单。

### 字段

- `渠道名称`

### 校验

- 不能为空
- 当前版本下不能重复

### 成功结果

中间区新增一个空渠道卡片，提示：

- 当前渠道还没有版本条目
- 点击 `新增版本条目`

## 4.4 新增渠道版本条目

### 入口

- 渠道卡片中的 `新增版本条目`

### 交互形式

右侧抽屉。

### 字段

- `渠道版本号`
- `发布日期`
- `ZIP 下载地址`
- `MD5`
- `更新说明`

### 更新说明交互

`release_note` 在 UI 中不直接暴露 JSON 数组，而是用“多行输入 + 列表预览”。

推荐交互：

- 一个多行文本框
- 用户每行输入一条说明
- UI 自动解析成数组预览

例如输入：

```text
更新 Aihelp SDK
优化红点未读消息逻辑
```

保存时写成：

```json
"release_note": [
  "更新 Aihelp SDK",
  "优化红点未读消息逻辑"
]
```

### 校验

- `version` 必填
- `release_date` 必填
- `zip_url` 必填且需符合 URL 格式
- `md5` 建议必填
- 当前渠道下 `version` 不能重复

## 4.5 编辑渠道版本条目

### 入口

- 表格行内的 `编辑`

### 行为

点击后：

- 右侧抽屉展开
- 带入当前条目已有数据
- 标题展示为：
  - `编辑 bilibili / 4.4.0`

### 保存后反馈

- 表格即时刷新
- 顶部状态从“未保存”恢复为“已保存”
- 给出轻量 toast：
  - `已保存 1.36.0 / bilibili / 4.4.0`

## 4.6 删除渠道版本条目

### 入口

- 行内 `删除`

### 交互

采用二次确认弹窗：

- 标题：`确认删除该渠道版本条目？`
- 描述：说明仅删除当前 OneSDK 版本下的这条记录

### 删除后效果

- 从表格即时移除
- 若该渠道已无条目：
  - 保留空渠道卡片，或者提示是否同时删除空渠道

推荐策略：

- 第一版删除条目后，如果渠道为空，自动保留空数组
- 不自动删除渠道对象，避免误伤

## 5. 保存与变更提示

## 5.1 保存模式

系统应采用“手动保存”模式，而不是自动保存。

原因：

- 这是配置型后台
- JSON 文件会直接写入工作区
- 自动保存风险更高

### 保存按钮状态

- 无修改：禁用
- 有修改：高亮可点击
- 保存中：loading
- 保存失败：红色错误状态

## 5.2 未保存提示

以下场景都要拦截：

- 切换 OneSDK 版本
- 刷新页面
- 关闭标签页

提示文案：

- `当前有未保存修改，确定离开吗？`

## 5.3 查看变更

顶部 `查看变更` 打开变更面板。

第一版不必做复杂 diff 视图，建议先做结构化摘要：

- `新增 OneSDK 版本 1.37.0`
- `新增渠道 lumo`
- `新增渠道版本 4.4.0`
- `修改 bilibili 4.4.0 的 zip_url`

后续再补 JSON diff。

## 6. 空状态设计

空状态要清晰区分 3 种情况：

### 6.1 仓库为空

- 没有任何 OneSDK 版本
- 提示用户点击 `新增 OneSDK 版本`

### 6.2 当前版本没有渠道

- 当前版本文件存在，但 `channels` 为空
- 提示用户新增渠道

### 6.3 当前渠道没有条目

- 渠道存在，但数组为空
- 提示用户新增渠道版本条目

## 7. 错误提示设计

错误提示需要分层，不要都做成一个红色 toast。

### 7.1 字段级错误

展示在表单项下方：

- `版本号格式不正确`
- `该渠道下已存在相同版本`
- `ZIP 地址格式不合法`

### 7.2 保存级错误

展示在表单顶部或全局错误条：

- `保存失败：version.schema.json 校验未通过`
- `保存失败：index.json 与版本文件路径不一致`

### 7.3 系统级错误

展示独立错误页或错误卡片：

- `无法读取 metadata 仓库`
- `无法写入目标文件`

## 8. 组件建议

为保证实现稳定，建议界面组件尽量标准化：

- 版本列表：侧边导航 + 简单统计
- 渠道区：卡片容器
- 条目区：标准表格
- 编辑器：抽屉表单
- 全局动作：顶部工具栏

不要第一版就追求复杂的可拖拽布局或自由编辑 JSON。

## 9. 权限与并发假设

当前产品默认假设：

- 使用人数很少
- 同时编辑概率低
- 不做复杂并发冲突解决

因此第一版只需要做到：

- 保存前重新读取一次目标文件
- 如果文件已被别人修改，提示用户刷新后重试

## 10. 推荐视觉风格

因为这是内部工具，视觉上不需要做营销化设计，但需要做到：

- 信息密度高
- 结构清晰
- 状态明确
- 编辑区和查看区区分明显

推荐风格：

- 浅色背景
- 明确分栏
- 表格和表单优先
- 成功、警告、错误状态颜色分明

## 11. 第一版必须具备的交互能力

第一版完成时，至少要满足：

1. 能浏览所有 OneSDK 版本
2. 能切换版本并查看渠道数据
3. 能新增 OneSDK 版本
4. 能新增渠道
5. 能新增和编辑渠道版本条目
6. 能校验并保存
7. 能提示未保存修改

## 12. 第二版可补的交互能力

后续可增强：

1. JSON diff 预览
2. git status 展示
3. git diff 展示
4. 保存历史
5. 删除 OneSDK 版本
6. 搜索和筛选

## 14. 编辑后进入发布的线性流程

这套后台不应该只停在“改完 JSON 并保存”，而应该把用户自然带到“确认并发布”。

推荐采用单主线流程：

```text
进入后台
  ↓
选择 OneSDK 版本
  ↓
查看当前渠道映射
  ↓
新增 / 编辑渠道版本条目
  ↓
点击保存
  ↓
系统完成字段校验 + schema 校验
  ↓
保存成功
  ↓
进入“发布确认”状态
  ↓
查看本次变更摘要
  ↓
确认执行发布
  ↓
执行 git 提交
  ↓
执行 git push
  ↓
发布完成
```

### 14.1 线性主界面状态

推荐把页面顶部状态条做成线性步骤条，而不是让用户自己猜当前走到了哪一步。

```text
[1 选择版本] → [2 编辑映射] → [3 保存校验] → [4 发布确认] → [5 发布完成]
```

状态规则：

- 当前步骤高亮
- 已完成步骤显示完成态
- 出错步骤显示错误态
- 未进入步骤保持灰态

## 15. 发布流程交互设计

### 15.1 保存后的状态切换

当用户点击 `保存` 且保存成功后，不应该只弹一个 toast 就结束，而应该出现明确的下一步引导：

```text
已保存当前修改
本次变更已写入 metadata 工作区
[查看变更] [进入发布]
```

这里的重点是：

- `保存` 解决的是“写入工作区”
- `发布` 解决的是“把工作区改动提交并推送”

这两个动作要区分，但要连续。

### 15.2 发布确认面板

点击 `进入发布` 后，右侧编辑区切换成“发布确认面板”。

建议展示以下内容：

- 本次修改涉及的文件
- 本次新增 / 修改 / 删除的映射摘要
- 当前 git 工作区状态
- commit message 输入框
- 操作按钮：
  - `返回继续编辑`
  - `提交 commit`
  - `提交并推送`

线框示意：

```text
┌──────────────────────────────────────────────┐
│ 发布确认                                      │
├──────────────────────────────────────────────┤
│ 本次变更摘要                                  │
│ + 新增 lumo / 4.13.0                         │
│ ~ 修改 bilibili / 4.4.0 的 zip_url            │
│                                              │
│ 涉及文件                                      │
│ - mappings/index.json                         │
│ - mappings/versions/1.36.0.json               │
│                                              │
│ Commit Message                                │
│ [ chore: update onesdk metadata ...        ] │
│                                              │
│ [返回继续编辑] [提交 commit] [提交并推送]      │
└──────────────────────────────────────────────┘
```

### 15.3 推荐的发布交互顺序

为了降低误操作，推荐把发布动作拆成两级：

1. `提交 commit`
2. `提交并推送`

不要第一版只保留一个“发布”大按钮。

原因：

- 有的人只想先落本地 commit
- 有的人希望先检查 diff 再 push
- 失败时更容易知道卡在 commit 还是 push

### 15.4 提交前校验

点击 `提交 commit` 或 `提交并推送` 前，需要做以下检查：

1. 当前没有未保存表单修改
2. 当前 metadata 文件校验通过
3. 当前 git 工作区只有允许提交的文件
4. commit message 不为空

如果任一项失败，不进入发布执行阶段。

### 15.5 发布执行态

发布过程中不要只显示一个 loading 圈，建议线性输出执行阶段：

```text
正在执行发布

1. 检查工作区状态        [完成]
2. 生成变更摘要          [完成]
3. 创建 git commit       [进行中]
4. 推送到远端            [等待中]
```

如果只点了 `提交 commit`，则第 4 步不执行。

### 15.6 发布完成态

发布成功后，页面进入成功状态：

```text
发布完成

提交信息：
- commit: abc1234
- branch: main

[继续编辑] [查看最新状态]
```

如果执行了 push，还应展示：

- 远端分支名
- push 是否成功

### 15.7 发布失败态

发布失败要准确区分失败阶段：

#### commit 失败

例如：

- commit message 为空
- git 用户信息未配置
- 工作区状态异常

界面提示：

```text
提交失败：git commit 未成功
原因：...
[返回修改] [重试]
```

#### push 失败

例如：

- 无远端权限
- 网络异常
- 远端分支被拒绝更新

界面提示：

```text
推送失败：git push 未成功
本地 commit 已创建，但远端未更新
[查看 commit] [重试 push] [返回]
```

这一点很重要，因为“commit 成功但 push 失败”是一个半成功状态，不能和完全失败混在一起。

## 16. 推荐最终主线

如果把产品交互收成一条最自然的主线，我建议是：

```text
选择 OneSDK 版本
  ↓
编辑渠道映射
  ↓
保存到工作区
  ↓
查看变更摘要
  ↓
提交 commit
  ↓
推送到远端
  ↓
发布完成
```

对应页面上的动作按钮也建议尽量线性：

- 编辑阶段：`保存`
- 保存后：`进入发布`
- 发布阶段：`提交 commit`
- commit 后：`提交并推送`

这样用户不会在一个页面上同时面对太多动作，流程也更符合“先编辑，再发布”的心理模型。
